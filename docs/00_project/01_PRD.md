## 2. 项目背景

珠海市迪奇孚瑞生物科技有限公司（Digifluidic）在数字微流控核酸检测领域积累了多年的技术成果，包括 VHunter Plus 全自动核酸分析仪、微流控芯片体系以及约 24 篇关键科研论文支撑的技术证据链。

当前公司需要将技术平台、产品体系、应用场景和论文证据链整理为可维护的对外宣传材料，用于客户介绍、展会路演、政府项目申报、合作伙伴沟通和内部培训等场景。

传统方式（PPT、Word、单次导出 PDF）存在以下问题：

1. 版本维护困难，每次更新需要重新排版；
2. 内容复用性差，同一信息需要在多个文件中重复维护；
3. 网页展示和 PDF 输出口径难以保持一致；
4. 论文证据链和应用矩阵的结构化程度低，不利于后续扩展；
5. 缺乏统一的合规表达约束机制。

因此，需要一个可在线预览、可导出 PDF、可版本化维护的静态网页宣传册项目。

---

## 3. 项目目标

1. 建立结构化宣传册内容体系，将公司技术平台、产品、应用场景和论文证据链整合为统一的数据模型；
2. 支持网页在线预览，提供清晰、专业的浏览体验；
3. 支持 A4 PDF 自动导出，确保网页和 PDF 内容一致；
4. 支持后续内容更新，通过修改 TypeScript 数据文件即可重新构建；
5. 支持内部评审流程，评审人员通过网页预览即可完成审查；
6. 支持 Windows Server + Nginx 静态部署，通过 8083 端口对内或对外提供访问；
7. 保持宣传表达合规，区分科研证据与产品声明，避免夸大或绝对化表述。

## 6. 使用场景

1. **客户拜访前生成或更新 PDF**：市场人员在拜访客户前，更新内容数据文件，重新构建并导出最新版 PDF；
2. **展会 / 路演材料准备**：在展会或路演前，生成最新版本宣传册，通过网页或 PDF 展示；
3. **内部评审网页预览**：技术、市场和合规负责人通过网页预览宣传材料，逐项审查文案、数据、论文引用和合规声明；
4. **政府项目申报材料辅助**：将宣传册作为项目申报的辅助材料，提供结构化的技术能力和应用场景说明；
5. **合作伙伴技术沟通**：在技术合作洽谈中，通过宣传册系统展示平台能力、方法学储备和应用矩阵；
6. **服务器部署后内网或公网访问**：将宣传册部署到 Windows Server，通过 8083 端口对内或对外提供访问；
7. **后续版本归档**：每次更新后归档对应版本的网页和 PDF，形成可追溯的版本历史。

## 7. 功能需求

### 7.1 宣传册网页展示

宣传册网页应包含以下模块：

1. **封面（CoverPage）**：项目名称、公司品牌标识、版本号、日期；
2. **执行摘要（ExecutiveSummary）**：公司和平台核心能力概述；
3. **平台概述（PlatformOverview）**：Digifluidic 数字微流控平台的技术定位与能力框架；
4. **产品生态（ProductEcosystem）**：VHunter Plus 分析仪、微流控芯片、试剂耗材的体系化说明；
5. **技术路线（TechnologyRoutes）**：LAMP、RT-qPCR、多重 PCR、电化学传感等方法学介绍；
6. **应用矩阵（ApplicationMatrix）**：应用方向 × 产品/材料 × 技术路线 × 表达重点的结构化矩阵；
7. **论文证据链（PaperEvidence）**：按应用方向组织的关键论文列表和证据说明；
8. **合规声明（ComplianceNotice）**：合规边界、表达规范和免责说明；
9. **页脚与版本信息（ContactAndFooter）**：联系方式、版本号、版权声明。

上述模块可作为 React 组件或页面区块实现，支持组件化复用和独立维护。

### 7.2 内容数据维护

宣传册内容应集中维护在 TypeScript 数据文件中，实现内容与视图分离：

1. 优先使用 TypeScript 数据文件（如 `src/data/brochure.ts`），便于类型约束和组件调用；
2. 后续如需多语言或非开发人员维护内容，可扩展为 JSON 配置文件；
3. 建议数据结构包括但不限于：

```
BrochureMeta：宣传册元信息（标题、版本号、生成日期、语言）
Metrics：关键指标数据（论文数量、应用方向数量、病原体覆盖范围等）
ApplicationItem：应用方向条目（方向名称、产品/材料示例、技术路线、表达重点）
PaperItem：论文条目（标题、作者、期刊、年份、DOI、对应应用方向、关键发现摘要）
ComplianceText：合规声明文本（合规说明、免责声明、推荐表达方式）
```

4. 页面组件从数据文件读取内容，不硬编码文案。

### 7.3 PDF 导出

1. **自动导出**：使用 Playwright 自动化导出 A4 PDF；
   - 本地启动 Vite 预览服务；
   - Playwright 打开指定 URL；
   - 等待页面渲染完成；
   - 使用 Chromium PDF 能力导出；
   - 保存到 `output/` 或 `exports/` 目录。
2. **手动兜底**：保留 Chrome 浏览器手动打印方案；
   - 打开本地或服务器页面；
   - 使用浏览器打印 → 另存为 PDF；
   - A4 页面尺寸，背景图形打开，页眉页脚关闭。
3. **文件命名规范**：推荐格式为 `Digifluidic_Brochure_customer_zh-CN_v2026.06.pdf`。
4. **PDF 质量检查**：导出后需检查封面完整性、页码正确性、页面断点、图表截断、表格跨页、中文字体、品牌色、合规声明和文件命名。

### 7.4 内部评审

1. 内部评审人员（技术、市场、管理、合规）通过网页预览审查宣传材料；
2. 评审内容包括：文案准确性、数据正确性、论文引用恰当性、合规声明完整性、页面视觉效果、PDF 导出质量；
3. 评审反馈后修改 TypeScript 数据文件或组件代码；
4. 重新构建、预览、导出 PDF，确认修改无误后记录版本变更。

### 7.5 静态部署

1. 构建产物为 `dist/` 目录，包含静态 HTML、JS、CSS 和资源文件；
2. 通过 Nginx 静态托管 `dist/` 目录；
3. Nginx 监听 8083 端口，使用独立 server block；
4. 不新增任何后端服务或 API；
5. 部署后验证网页访问和 PDF 导出功能。

### 9.1 应包含的内容

1. 公司与品牌基本信息；
2. VHunter Plus 全自动核酸分析仪介绍；
3. 微流控核酸检测芯片体系；

## 12. 部署需求

1. **部署目录**：推荐 `C:\digifluidic-brochure`；
2. **WebRoot**：`C:\digifluidic-brochure\dist`；
3. **Nginx 配置**：在 Nginx 中新增独立 server block，监听 8083 端口；
4. **路由配置**：使用 `try_files $uri $uri/ /index.html` 确保前端路由正常工作（如为单页无路由也可保留）；
5. **端口隔离**：不修改已有系统的端口和 server block；
6. **配置测试**：修改 Nginx 配置后必须先执行 `nginx -t` 测试，确认无误后再 reload；
7. **部署验证**：部署后需验证浏览器访问（`http://服务器IP:8083`）和 PDF 导出功能；
8. **Windows 路径**：Nginx 配置中建议使用正斜杠（`/`）表示路径。

## 14. 合规与传播边界

1. 本项目生成的材料用于宣传介绍、内部评审、客户沟通和项目申报辅助；
2. 宣传材料不替代产品注册文件、检测性能声明、法定说明或正式技术协议；
3. 文献内容仅用于说明技术研究基础、应用方向和证据链，不应表述为具体产品性能承诺；
4. 具体产品功能、性能指标、适用范围和注册状态以公司正式产品资料、注册文件或经审批的对外材料为准；
5. 对外发布前，材料需经过公司内部技术、市场或合规负责人确认；
6. 在宣传文案中应避免以下表述：
   - 权威证明
   - 完全验证
   - 绝对领先
   - 保证检测
   - 唯一方案
   - 全球第一
   - 零风险
   - 百分百准确
   - 全面替代传统检测
   - 已完全覆盖所有病原体
   - 临床金标准
7. 推荐使用以下表达方式：
   - 研究显示
   - 文献报道
   - 在相关研究中
   - 可用于支持某类应用场景的技术基础
   - 展示了平台在某方向的应用潜力
   - 有助于构建多靶标检测能力
   - 可作为现场检测方案的技术参考
   - 具体性能以正式产品资料和注册文件为准

## 16. 后续扩展方向

以下方向不作为当前版本需求，仅供未来参考：

1. **多语言支持**：扩展为中文、英文等多语言版本，通过 i18n 或分语言数据文件实现；
2. **多版本宣传册**：支持维护面向不同受众的多版本宣传册（如客户版、政府版、技术版）；
3. **JSON 内容配置**：将 TypeScript 数据文件迁移为 JSON 格式，便于非开发人员直接编辑；
4. **更丰富的图表**：引入图表库展示技术路线图、应用覆盖热力图等可视化内容；
5. **自动化 PDF 批量导出**：支持一键导出所有语言/版本组合的 PDF；
6. **CMS 接入可能性**：后续如需非技术人员维护内容，可考虑接入轻量 CMS，但当前阶段不做；
7. **文献数据结构化管理**：将论文证据链进一步整理为结构化数据文件或轻量数据源，支持更灵活的检索和引用；如未来确有必要，再评估数据库化方案。

以上扩展方向仅作为未来设想，不进入当前版本的功能范围或验收标准。
