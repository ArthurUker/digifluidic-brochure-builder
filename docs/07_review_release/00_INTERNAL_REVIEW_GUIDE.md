## 2. 文档目的与适用范围

本文件用于：

1. 定义白皮书/宣传册的内部评审流程；
2. 统一技术评审、内容评审和合规评审的标准；
3. 确保对外发布前通过必要的内部确认。

本文件不替代：

1. 发布流程（`01_RELEASE_PROCESS.md`）；
2. 版本号规则（`02_VERSIONING_RULES.md`）；
3. 公司正式审批流程或法务审核意见。

## 3. 评审类型与参与角色

### 3.1 评审类型

| 评审类型 | 说明 | 时机 |
|---|---|---|
| **技术评审** | 代码、构建、部署和 PDF 导出技术实现 | 前端骨架完成后、部署前 |
| **内容评审** | 文案准确性、表达口径、数据正确性 | 内容数据编写完成后 |
| **合规评审** | 合规声明完整性、表达边界、论文引用合规性 | 内容定稿后、对外发布前 |
| **设计评审** | 页面视觉、品牌规范、打印效果 | 设计定稿后 |
| **PDF QA** | PDF 输出质量 | 每次 PDF 导出后 |

### 3.2 参与角色

| 角色 | 评审职责 |
|---|---|
| 技术负责人 | 技术评审、构建和部署验证 |
| 内容/市场负责人 | 内容评审、文案准确性和表达口径 |
| 合规/法务负责人 | 合规评审、声明完整性和表达边界 |
| 设计负责人 | 设计评审、品牌规范遵守情况 |
| 项目经理/协调人 | 评审流程组织和结果汇总 |

## 4. 评审流程

### 4.1 标准流程

```
内容编写/代码开发完成
  ↓
自检（开发者/内容编写者自查）
  ↓
技术评审
  ↓
内容评审
  ↓
设计评审
  ↓
合规评审
  ↓
PDF QA
  ↓
问题修复
  ↓
复评（针对修复内容）
  ↓
全部通过 → 批准发布
```

### 4.2 轻量流程（小改动）

如仅修改了错别字、联系方式等非关键内容：

```
修改完成
  ↓
自检
  ↓
内容负责人确认
  ↓
PDF QA（快速检查）
  ↓
发布
```

### 4.3 评审记录

每次评审应记录：

- 评审日期；
- 评审类型；
- 参与人员；
- 评审版本号；
- 通过/不通过项；
- 修改建议；
- 修复确认。

## 5. 技术评审标准

### 5.1 代码质量

| 检查项 | 通过标准 |
|---|---|
| TypeScript 类型 | 无类型错误（`tsc --noEmit` 通过） |
| 构建 | `npm run build` 成功 |
| 代码风格 | 符合 `docs/04_development/02_CODE_STYLE.md` |
| 组件规范 | 符合 `docs/04_development/03_COMPONENT_SPEC.md` |
| 无范围外依赖 | 未引入后端、数据库、CMS 等技术 |

### 5.2 构建产物

| 检查项 | 通过标准 |
|---|---|
| dist/ 完整 | `index.html` + `assets/` 存在 |
| 本地预览 | `npm run preview` 页面正常 |
| 文件大小 | dist/ 总大小合理；如明显偏大，应检查图片压缩、未使用资源和构建产物体积 |

### 5.3 PDF 导出

| 检查项 | 通过标准 |
|---|---|
| Playwright 导出 | `npm run export:pdf` 成功 |
| Chrome 打印 | 手动打印正常 |
| QA 清单 | 按 `docs/06_pdf_export/03_PDF_QA_CHECKLIST.md` 通过 |

## 6. 内容评审标准

### 6.1 文案准确性

| 检查项 | 通过标准 |
|---|---|
| 公司名 | 与工商注册名称一致 |
| 产品名 | 与公司正式产品命名一致 |
| 技术术语 | 使用正确、无歧义 |
| 数据 | 有依据、可溯源 |
| 日期 | 版本号、发布日期正确 |

### 6.2 表达口径

| 检查项 | 通过标准 |
|---|---|
| 无绝对化表达 | 无"全球领先""唯一""100%"等禁用词 |
| 无夸大宣传 | 表达客观、有依据 |
| 合规推荐表达 | 使用"研究显示""在相关研究中"等推荐表达 |
| 论文引用合规 | relevance 字段使用合规表达 |

### 6.3 内容完整性

| 检查项 | 通过标准 |
|---|---|
| 白皮书主线 | 10 个主线章节完整 |
| 应用矩阵 | 数据完整、阶段标注正确 |
| 论文证据 | 列表完整、分组正确 |
| 合规声明 | 全局声明和章节声明完整 |

## 7. 合规评审标准（关键）

### 7.1 声明完整性

| 检查项 | 通过标准 |
|---|---|
| 全局合规声明 | 白皮书末尾有完整声明 |
| 论文证据声明 | 每组论文末尾有声明 |
| 应用矩阵声明 | 矩阵表格下方有声明 |
| 行内声明 | 产品参数附近有声明（如有） |

### 7.2 表达边界

| 检查项 | 通过标准 |
|---|---|
| 论文 ≠ 产品承诺 | 无将论文结果写成产品性能承诺的表述 |
| 科研 ≠ 注册 | 区分了技术储备和注册产品 |
| 无未授权认证 | 无未经确认的认证、注册或审批声明 |
| 声明可读 | 合规声明视觉上清晰可辨 |

### 7.3 红线检查

以下情况为合规红线，必须驳回：

1. 声称产品已获某监管机构认证但无法提供正式文件；
2. 将论文灵敏度/特异性等数据直接作为产品性能参数；
3. 使用"经XX权威机构认证"但无法提供认证文件；
4. 合规声明被弱化处理（小字、浅色、位置隐蔽）。

## 8. 设计评审标准

| 检查项 | 通过标准 |
|---|---|
| 品牌规范 | Logo、品牌色、字体符合 `docs/03_design/03_BRAND_ASSETS_GUIDE.md` |
| 页面布局 | 符合 `docs/03_design/01_PAGE_LAYOUT_SPEC.md` |
| 打印效果 | 打印预览正常，分页合理 |
| 可读性 | 字号、行距、对比度合理 |

## 9. 评审结果处理

### 9.1 通过

- 该评审类型标记为"通过"；
- 进入下一评审环节或批准发布。

### 9.2 有条件通过

- 存在不影响核心内容的轻微问题；
- 记录问题并限期修复；
- 可先进入下一环节，但发布前需确认修复。

### 9.3 不通过

- 存在合规红线、内容错误或技术故障；
- 必须修复后重新提交评审；
- 阻塞后续环节。

## 10. 当前阶段不纳入

1. 自动化评审工具；
2. 评审系统或平台；
3. 外部评审流程（客户评审、监管评审等）；
4. 评审 SLA 要求。

## 11. 风险与控制措施

| 风险 | 控制措施 |
|---|---|
| 合规评审被跳过 | 发布流程中合规评审为必须环节，不可跳过 |
| 评审流于形式 | 使用本文件的检查清单逐项确认 |
| 评审意见未落实 | 复评环节确认修复情况 |
| 评审职责不清 | 本文件第 3.2 节明确了各角色职责 |
