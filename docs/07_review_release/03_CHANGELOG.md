## 2. 文档目的与适用范围

本文件用于：

1. 记录白皮书/宣传册每个版本的变更内容；
2. 帮助团队成员和后续维护者了解版本演进历史；
3. 作为内部评审和对外沟通的参考。

本文件不替代：

1. Git 提交记录（代码级变更）；
2. 评审记录（评审意见和结论）。

## 2.1 版本号格式说明

| 阶段 | 版本号格式 | 示例 | 说明 |
|---|---|---|---|
| 开发阶段（未正式发布） | `v0.x-draft` | `v0.1-draft` | 文档主干建设和前端骨架阶段使用 |
| 正式发布版本 | `v年份.月份` | `v2026.06` | 第一次对外发布后切换为此格式 |
| 同月多次发布 | `v年份.月份.序号` | `v2026.06.2` | 同月第二次及以后发布时追加序号 |

> 版本号格式详细规则见 `02_VERSIONING_RULES.md`。

## 3. 变更记录格式

### 3.1 版本条目模板

```markdown
## [版本号] - YYYY-MM-DD

### 新增
- 新增内容描述

### 变更
- 变更内容描述

### 修复
- 修复内容描述

### 技术变更
- 技术层面变更描述

### 评审
- 评审日期和结论
```

### 3.2 变更类型说明

| 类型 | 说明 | 示例 |
|---|---|---|
| **新增** | 新增的内容、功能或章节 | "新增食品安全应用场景章节" |
| **变更** | 修改或调整已有内容 | "更新公司联系方式" |
| **修复** | 修复错误或问题 | "修正产品名称拼写错误" |
| **技术变更** | 代码、样式、构建或部署相关变更 | "优化 PDF 导出分页逻辑" |
| **评审** | 评审信息和结论 | "2026-06-XX 通过内部评审" |

## 5. 变更记录维护规则

### 5.1 何时追加

每次发布新版本时必须追加变更记录，包括：
- 正式对外发布；
- 内部预览版发布；
- 重要里程碑版本。

### 5.2 谁来维护

- 由发布执行人（开发者或项目经理）在发布时更新；
- 变更内容由各评审负责人确认。

### 5.3 维护原则

1. 最新版本在最上方；
2. 每个版本条目独立；
3. 变更描述简洁清晰，面向非技术人员也可理解；
4. 技术变更与技术评审结论分开记录。

## 6. 当前阶段不纳入

1. 自动化 CHANGELOG 生成；
2. 与 Git 提交记录自动关联；
3. 多语言 CHANGELOG。
